# Architecture

Current-state architecture for `agent-loom`.
This document reflects what is implemented in the repo on 2026-04-10. It distinguishes live repo structure from draft or evolving design ideas.

## Overview

`agent-loom` is a cross-platform skill library built around the `agentskills.io` model. The repo has two main architectural concerns:

1. The skill library itself: reusable skills stored in `.agents/skills/`, plus the docs and control files that keep the library consistent.
2. The process-and-agent design layer: a newer planning layer that reasons about process structure before execution and feeds execution into `project-orchestrator`.

The repo is not a full agent harness. It is a portable skill layer that can be globally installed into multiple AI coding tools, or copied into project-local skill folders where needed.

## Top-Level Structure

| Path | Responsibility |
|------|----------------|
| `.agents/ROUTING.md` | Global routing and conflict-resolution rules for skill selection |
| `.agents/skills/<name>/SKILL.md` | Individual skill entry points |

| `docs/specs/` | Design specs and architecture proposals |

| `README.md` | User-facing install and usage guide |
| `install.sh`, `install.ps1` | Global installation via symlink/junction setup |
| `uninstall.sh`, `uninstall.ps1` | Removal of repo-managed global installs |

## Architectural Layers

### 1. Distribution Layer

The repo is distributed as a source-of-truth skill library. Installation is file-based:

- the repo is cloned once
- installer scripts create symlinks or junctions into tool-recognized global skill directories
- tools then load the same skill definitions from the repo checkout

This keeps updates simple: `git pull` plus install refresh updates the active skill library across supported tools.

### 2. Control Layer

Two files govern how the library behaves as a system:

- `.agents/ROUTING.md`
  defines routing precedence and the process/agent-design firing order
- `docs/SKILL-INDEX.md`
  defines the skill catalog, categories, and high-level call graph

These are the repo’s coordination documents. They do not execute work, but they define how agents should choose and combine skills.

### 3. Skill Layer

The core of the repo is `.agents/skills/`. Skills are organized by role rather than by programming language:

- Meta skills manage the library itself
- Thinking skills provide structured reasoning methods
- Project-specific skills produce project outputs or coordinate execution
- Security skills gate any handling of external content

Each skill is expected to remain compact and act as an entry point, with references moved to sibling files where needed.

### 4. Process-and-Agent Design Layer

This layer is now present in the repo as implemented skills, not just a spec. It adds planning above direct skill execution:

- `process-decomposer`
  checks for reusable process patterns, classifies complexity, and writes process entries to `docs/processes/`
- `agent-builder`
  designs execution structure from a process entry and, by current skill contract, writes architecture specs to `docs/architecture/`
- `setup-evaluation`
  validates decomposition and architecture before execution
- `project-orchestrator`
  remains the execution wrapper and feedback owner

This layer exists to avoid treating every request as a fresh one-off execution problem.

## Execution Architecture

The repo’s current execution model is:

1. A request is routed by intent.
2. If the request is simple, it can go directly to a single skill.
3. If the request is complex, `process-decomposer` runs first.
4. `process-decomposer` classifies the work as:
   - `single-skill`
   - `skill-chain`
   - `agent-chain`
   - `exact-match`
5. For `skill-chain`, execution still goes through `project-orchestrator`.
6. For `agent-chain`, `agent-builder` writes an architecture spec, then `setup-evaluation` validates it, then `project-orchestrator` executes it.
7. After execution, `project-orchestrator` writes execution feedback back into the process entry.

That last step is important: the process registry is intended to improve by replay and write-back, not just by storing static plans.

## Current Data and Artifact Flow

### Skill Library Flow

`README.md` / install scripts
→ global or project-local install
→ tool loads skill from `.agents/skills/`
→ routing/index docs guide selection

### Process-Reuse Flow

User request
→ `process-decomposer`
→ `docs/processes/process.md` lookup
→ new or reused process entry
→ `agent-builder` if needed
→ `setup-evaluation` if agent-chain
→ `project-orchestrator`
→ execution
→ execution feedback written back to process entry

### Documentation Flow

Design and product artifacts are stored under `docs/`, while generated-file activity is logged in `docs/skill-outputs/SKILL-OUTPUTS.md`.

## Key Components

### `project-orchestrator`

This is the repo’s execution coordinator. In the current architecture it:

- routes requests to the right skill
- handles process-backed execution
- waits for architecture and setup validation where required
- owns the execution feedback loop for process entries

It is the operational bridge between planning and execution.

### `process-decomposer`

This is the pre-execution reasoning entry point for complex work. It owns:

- process lookup
- complexity triage
- outcome-first decomposition
- process entry creation in `docs/processes/`

It plans, but does not execute.

### `agent-builder`

This component turns a decomposed process into an execution structure. It owns:

- deciding whether no agent, one agent, or multiple agents are needed
- defining topology and boundaries
- persisting architecture specs according to the skill contract

It designs, but does not execute.

### `setup-evaluation`

This is the pre-execution QA gate for agent-chain work. It checks:

- decomposition quality
- architecture correctness
- step coverage and agent coverage
- linkage consistency between process and architecture artifacts

It evaluates, but does not modify.

### Security Skills

The `secure-*` skills form a hard boundary around external content handling. Architecturally, they are not optional helpers; they are a mandatory enforcement layer with higher priority than ordinary repo content.

## Boundaries and Non-Goals

This repo currently is:

- a portable skill library
- a routing and orchestration system for those skills
- a process-memory and pre-execution reasoning layer

This repo currently is not:

- a standalone execution runtime
- a complete enterprise agent harness
- a general-purpose workflow engine outside the skill model

## Current State vs Design Intent

The repo now contains both:

- implemented skill-layer behavior
- design-spec documents for why that behavior exists and how it should evolve

The most important distinction is that `docs/specs/` contains design intent, while `.agents/skills/`, `.agents/ROUTING.md`, `docs/SKILL-INDEX.md`, and `docs/processes/` represent the live architecture. Some artifact locations, such as `docs/architecture/`, are currently defined by skill contracts even if they are not yet populated in this checkout.

## Practical Mental Model

The simplest accurate mental model for this repo is:

`agent-loom` is a globally installable skill library with a control plane for routing and a newer planning layer for reusable process decomposition and agent-structure design before execution.
